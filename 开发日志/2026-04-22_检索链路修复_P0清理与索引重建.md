# 2026-04-22｜检索链路修复（P0：清理污染 + 重建 FAISS）

## 背景（问题复现）

- **现象**：查询“残差网络”召回到 `README.md` 广告、`XGBoost.md` 等不相关内容。

- **`docs/` 现状**：`docs/` 内包含大量 `*.pdf` 与 `README.md` 等非检索目标文件，极易造成**索引污染**与召回稀释。

## Step 1：清理 `docs/` 污染（删 README/PDF/HTML/广告）

### 1.1 遇到的问题

- **zsh 的 `nomatch`**：直接执行 `rm -f docs/*.pdf` 在“无匹配文件”时会触发 `nomatch` 报错并中断，导致清理不完整。

### 1.2 修正后的清理方式

- **做法**：用 `zsh -lc 'setopt nonomatch; ...'` 临时关闭 `nomatch`，保证通配符在无匹配时不会中断脚本。
- **效果验证**：清理后统计：
  - `pdf: 0`
  - `html: 0`
  - `readme: 0`

## Step 2：按主题预留子目录（空间隔离）

- 同步尝试按文件名关键词移动部分文件到对应目录（如 transformer/attention/llm、xgboost/决策树等）。

> 备注：这一步的核心价值是“**后续可控维护**”，不是立刻提升召回（提升召回主要来自更干净的索引与更强的召回策略）。

## Step 3：重建 FAISS 索引（标题/段落优先切分）

- **索引输出路径**：`./faiss_index`
- **输入文档**：`docs/` 下所有 `*.md/*.txt`（忽略空文本）
- **切分策略**：`chunk_size=500`、`chunk_overlap=50`，优先按：
  - `\n\n`（段落/标题边界）
  - `\n`
  - `。`、`；`

- **重建结果**：

## 2026-04-22 11:18｜评估与测试集：发现的结构性问题与修复

### 1) `evaluation_report.json` 仍出现 `README.md`

- 说明：你的 `test_set.json` 里还保留着旧的 `README.md` 来源条目；即使你清理了 `docs/README*`，评估依旧会按 `test_set.json` 的 `source` 字段展示为 `README.md`。
- 建议：在清理语料/重建索引后，需要**重新生成测试集**，避免“测试集与语料不一致”。

### 2) `evaluate_batch.py` 与 `app.py` 检索链路不一致

- 旧实现：`evaluate_batch.py` 使用 **纯向量** `vectorstore.as_retriever(k=3)`；而你现在前端已经是 **Hybrid（向量+BM25）+ CrossEncoder rerank**。
- 修复：将 `evaluate_batch.py` 的粗召回切换为 Hybrid（向量+BM25），并把 `context` 写成带 `source` 的格式，保证评估更接近线上链路。

### 3) `generate_test_set_batch.py` 只扫 `docs/*.md` 顶层

- 风险：你已经把部分笔记放进 `docs/dl_nlp/` 等子目录；若测试集生成只看顶层，会导致**生成覆盖不足**、成功率/代表性下降。
- 修复：改为递归扫描 `docs/**/*.md`，并用相对路径作为 `source`（例如 `dl_nlp/transformer-课堂笔记.md`）。

### 4) “并行问答 vs batch 生成”导致 epoch 失败的原因判断

- 现象解释：前端提问与 batch 生成同时打到同一个 `Ollama /api/generate`，会产生资源竞争（同模型单机并行时更易出现超时、截断、输出非 JSON），从而触发“JSON 解析失败”。
- 结论：**不建议并行**运行网页前端与测试集生成/评估脚本；或至少分离模型/端口/进程。
- 缓解：在 batch 脚本中增加重试与超时（已加入 3 次重试 + timeout），并把温控/休息保留，提升成功率。

## 2026-04-22 11:45｜P2：测试集过滤 + 召回/上下文调参 + 前端与 batch 互斥

### 1) 测试集生成阶段过滤“不可回答/过细节”

- 改动文件：`generate_test_set_batch.py`
- 策略：弱监督“可回答性”过滤（evidence filter）
  - `question` 必须非空
  - `answer_points` 必须为非空 list，且长度不超过 6（防止过碎/过细）
  - 每个要点截取前 12 个字符作为“短证据片段”，要求在对应文档片段中命中
  - 通过阈值：至少命中一半要点（向上取整）
- 目的：减少“文档未直接提及/没有实现细节”造成的评估噪声，让指标更可信。

### 2) Hybrid 召回候选池增大 + rerank 后 top_n 提升

- 改动文件：`app.py`、`evaluate_batch.py`
- 参数：
  - `vec_k/bm25_k`: 15 → **25**
  - `rerank top_n`: 3 → **5**
- 目的：提高召回覆盖与回答完整性，同时依赖 rerank 控制噪声。

### 3) 互斥运行：batch 脚本运行时禁用前端，前端运行时禁止 batch

- 新增：`core/run_lock.py`
- 机制：本地锁目录 `.coderag_locks/`
  - `frontend.json`：前端定期刷新（Streamlit rerun 触发 touch）
  - `batch.json`：生成/评估脚本 acquire + 周期 refresh + finally release
- 行为：
  - batch 检测到活跃前端锁会直接退出并提示（避免并行抢 Ollama）
  - 前端检测到活跃 batch 锁会进入只读模式（禁用导入/对话入口）
