# 2026-04-22｜覆盖率优化第二阶段：大候选池、HyDE、去噪的消融试验

> 背景：上一阶段覆盖率 56.1% 是基于旧测试集（22条）。测试集更换后（13条），覆盖率暴跌至 34.6%。本阶段目标是找回并突破此前的覆盖率水平。

---

## 一、已执行的优化措施

### 1. 重建 FAISS 索引

- **原因**：`docs/` 目录有增删，索引与文档不一致是检索跑偏的首要嫌疑。
- **结果**：100 个文档 → 923 个 chunks，baseline 覆盖率从 19.2% → 23.1%。

### 2. 增大候选池 + 精排数量

| 参数 | 旧值 | 新值 |
|------|------|------|
| `vec_k` | 20 | **40** |
| `bm25_k` | 20 | **40** |
| `rerank_k` | 5 | **10** |

- **结果**：hybrid_rerank 覆盖率从 34.6% → **42.3%**（+7.7%）
- **代价**：检索延迟从 ~600ms → ~920ms（CrossEncoder 计算量翻倍）

### 3. 改进覆盖率计算：模糊匹配

- **旧逻辑**：`answer_point.lower()[:10] in context.lower()`
- **新逻辑**：双重匹配
  1. 去除空格/标点后的模糊匹配（处理公式类 `loss = r(A)` vs `loss=r(A)`）
  2. 原始前10字符 fallback（处理中英文表述差异）
- **效果**：公式类问题的匹配成功率提升，但受限于检索召回，总体覆盖率提升有限。

### 4. 实现 HyDE（Hypothetical Document Embeddings）

- **做法**：查询时先用 Ollama 生成假设答案，用假设答案做向量检索，BM25 仍用原始 query。
- **结果**：覆盖率 **42.3%**，与 hybrid_rerank **完全相同**，无额外增益。
- **延迟**：检索延迟从 ~920ms 暴增至 **~16,000ms**（每个 query 额外调用 Ollama）。

**HyDE 无效的原因分析**：
1. 当前测试集问题过于具体（如"lora 中的 d*d 2*r*d 公式代表什么含义？"），Ollama 生成的假设答案与原始 query 的 embedding 差异极小。
2. 假设答案质量有限（qwen2.5:7b 对公式的表达能力不强），未能生成更接近文档表述的假设文本。
3. 本地模型生成假设答案的速度太慢，性价比极低。

### 5. 实现上下文去噪（ContextDenoisePlugin）

- **做法**：精排后，按句子切分，计算每句与 query 的 embedding 相似度，只保留高相似度句子。
- **阈值**：0.55，最多保留 8 句，最少保留 3 句。
- **结果**：覆盖率从 42.3% 暴跌至 **16.7%**（单独去噪）和 **14.1%**（窗口+去噪）。

**去噪有害的原因分析**：
1. `all-MiniLM-L6-v2` 对公式/代码的语义理解极差，包含正确答案的句子（如公式推导）与 query 的相似度低于阈值，被错误过滤。
2. 最少保留 3 句的策略过于激进，对于答案分散在多个句子的场景，3 句不足以覆盖全部要点。
3. 去噪后 LLM 评分准确性反而上升（3.3 vs 2.8），说明去噪确实减少了噪声——但同时把**信号**也过滤掉了，导致覆盖率崩盘。

## 三、核心洞察

### 洞察 1：当前瓶颈已从"检索策略"转向"Embedding 模型能力"

- 大候选池已经把覆盖率从 23% 拉到 42%。
- 但后续所有插件（窗口、去噪、HyDE）都无法突破 42.3%。
- 这说明：**在当前 embedding 模型（all-MiniLM-L6-v2）的能力边界内，42% 已经接近上限。**
- 要继续提升，必须升级 embedding 模型，或者改变 chunk 粒度。

### 洞察 2：去噪插件的"准确性上升、覆盖率暴跌"悖论

| 指标 | 无去噪 | 有去噪 | 变化 |
|------|--------|--------|------|
| 覆盖率 | 42.3% | 16.7% | -25.6pp |
| 准确性 | 2.8 | 3.3 | +0.5 |
| 相关性 | 3.5 | 4.2 | +0.7 |

- 去噪后，context 更"干净"了，所以 LLM 评分认为"相关性更高、幻觉更少"。
- 但干净的同时，**正确答案也被洗掉了**。
- **结论**：当前的去噪策略是"宁可错杀，不可放过"，在 RAG 场景下不可接受。

### 洞察 3：HyDE 对具体问题无效

- HyDE 的核心假设：原始 query 的 embedding 与文档有语义鸿沟，需要假设答案来桥接。
- 但当前测试集的问题本身就很具体（如公式、代码），假设答案和原始 query 的 embedding 差异极小。
- HyDE 更适合**抽象/开放性问题**（如"解释一下 Transformer 的原理"），而非**具体/细节问题**（如"lora 公式含义"）。

## 四、零覆盖率问题根因

13 条测试集中，以下 6 题在所有配置下均为 0% 或接近 0%：

| 问题 | 类型 | 根因判断 |
|------|------|---------|
| 如何计算指令微调的损失函数？ | 公式 | 文档中有公式，但向量和 BM25 都未能召回 |
| 多设备加速比是如何定义的？ | 概念+公式 | 同上 |
| 如何通过逻辑路由选择正确的索引？ | 代码/实现 | 文档中有相关内容，但检索未命中 |
| lora 算法中的 h= Wor + AW 如何表示？ | 公式 | 测试集答案有乱码，文档中公式表述差异大 |
| lora 中的 d*d 2*r*d 公式代表什么含义？ | 公式 | 答案只是重复公式，无"含义"解释 |
| 在进行大语言模型调用时，如何确保响应文本的正确性？ | 代码步骤 | 文档中有相关步骤，但检索未命中 |

**共同特征**：
1. 都是**公式/代码/具体实现细节**类问题
2. `all-MiniLM-L6-v2`（384维，英文预训练为主）对中文公式/代码的语义表示能力弱
3. BM25 + jieba 分词对公式/代码的分词效果差（`d*d` → `d` `*` `d`）

## 五、下一步真正有效的措施

### P0：换 Embedding 模型（最可能有突破）

| 模型 | 大小 | 维度 | 预期效果 |
|------|------|------|---------|
| `bge-small-zh` | ~100MB | 512 | 中文优化，公式/代码理解更好 |
| `bge-base-zh-v1.5` | ~400MB | 768 | 更强的中文语义能力 |
| `bge-m3` | ~2GB | 1024 | 多语言+长文本，效果最强 |

- **推荐**：先试 `bge-small-zh`（100MB，本地无压力）
- **预期**：覆盖率从 42% → 55%~65%
- **代价**：需重建 FAISS 索引

### P1：改进 BM25 公式/代码分词

- 对英文/数字/符号混合文本（如公式、代码），保留原始 token 不被 jieba 拆开。
- 示例：`d*d` 不应被拆成 `d` `*` `d`，而应作为整体 `dd` 或保留原样。

### P1：增大 chunk_size

- 当前 1500，可试 2000~2500。
- 减少 chunk 边界对公式/代码的切割，让更多完整公式留在同一个 chunk 中。
- **代价**：需重建索引，chunk 变长后可能混入更多噪声。

### P2：修复测试集质量

- "lora 算法中的 h= Wor + AW" 答案要点2是乱码 `- e W SRAFY | ERS,`
- "lora 中的 d*d 2*r*d 公式代表什么含义？" 答案只是重复公式，无含义解释
- 测试集本身质量差，导致覆盖率被系统性压低

### ❌ 已证明无效/有害的措施（应避免）

- HyDE（qwen2.5:7b 本地生成假设答案）：无效且极慢
- ContextDenoisePlugin（阈值 0.55）：有害，过滤掉正确答案
- ContextExpansionPlugin：与 SentenceWindow 打平，冗余

## 七、结论

**当前最优配置（已锁定）**：

```python
hybrid = HybridRetriever(vectorstore, chunks, vec_k=40, bm25_k=40)
rr = RerankRetriever(hybrid, k=10)  # 无插件
```

- **覆盖率**：42.3%（13条测试集）
- **相关性**：3.5/5
- **检索延迟**：~920ms

**要突破 42%，必须升级 Embedding 模型**。在当前 `all-MiniLM-L6-v2` 的能力边界内，所有后处理插件都无法带来增益。
