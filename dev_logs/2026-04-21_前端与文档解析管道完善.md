# 2026-04-21｜前端与文档解析管道完善

> 目标：修复 Streamlit 前端缓存问题，提升 PDF 解析质量，将向量库从 Chroma 迁移至 FAISS，并接入 CrossEncoder 精排。
>
> 技术栈：Python, Streamlit, FAISS, pymupdf4llm, sentence-transformers, Ollama

---

## 一、问题诊断

### 1.1 Streamlit 前端报错

| 问题 | 现象 | 根因 |
|------|------|------|
| 缓存冲突 | `RuntimeError: Cannot send a request, as the client has been closed` | `@st.cache_resource` 与 HuggingFace Hub 的 httpx HTTP 客户端生命周期冲突，HF Hub 内部客户端被序列化/反序列化后失效 |
| 重复初始化 | sidebar 的 markdown 里直接调用 `init_vectorstore()` | 每次重渲染都触发 HuggingFace 连接 |

### 1.2 PDF 解析质量差

对比 PDF 原文与生成 MD：

| 丑陋表现 | 原因 | 修复方向 |
|---------|------|---------|
| 全是 `\n` 换行符，无段落结构 | `page.get_text()` 返回原始换行 | 按文本块提取，保留阅读顺序 |
| 公式变乱码 | PDF 内嵌字体子集 | 用 `pymupdf4llm` 替代原生 fitz |
| 代码块缩进丢失 | 空格被当成普通字符处理 | 手动保留 leading spaces |
| 页眉页脚混入正文 | 没有过滤边缘区域 | 按 bbox 坐标过滤 |
| 表格变成一坨 | PDF 表格无结构化 | 用专门工具提取表格 |
| 图片完全丢失 | `get_text()` 不提取图片 | 图片占位符标记 |

### 1.3 向量库选型回退

| 向量库 | 状态 | 原因 |
|--------|------|------|
| Chroma | ❌ 放弃 | SQLite 只读锁定，并发重建时出问题 |
| **FAISS** | ✅ **采用** | 纯文件存储，无锁，更适合本地离线场景 |

## 二、修复措施

### Step 1：修复 app.py 缓存问题

**方案**：`@st.cache_resource` 只缓存 embedding 模型，不缓存 Chroma 客户端。

```python
@st.cache_resource
def get_embedding():
    """只缓存 embedding 模型，不缓存向量库客户端"""
    return HuggingFaceEmbeddings(
        model_name="all-MiniLM-L6-v2",
        model_kwargs={"local_files_only": True}  # 强制离线，不连 HF Hub
    )

def init_vectorstore():
    """每次调用新建连接，但复用已缓存的 embedding"""
    if not os.path.exists("./chroma_db") or not os.listdir("./chroma_db"):
        return None
    embedding = get_embedding()
    return Chroma(persist_directory="./chroma_db", embedding_function=embedding)
```

sidebar 状态显示改为只检查目录存在性，不直接调用 `init_vectorstore()`：

```python
db_exists = os.path.exists("./chroma_db") and os.listdir("./chroma_db")
st.markdown(f"向量库：{'已加载' if db_exists else '未创建'}")
```

### Step 2：升级 ingest.py PDF 解析

**核心改进**：换用 `pymupdf4llm`（专为 LLM/RAG 设计的 PDF→Markdown 工具）。

安装依赖：

```bash
pip install pymupdf4llm
```

关键改动点：

| 改动 | 效果 |
|------|------|
| `pymupdf4llm.to_markdown()` 替代 `page.get_text()` | 自动识别标题层级、表格转 MD、代码块保留缩进 |
| 自动过滤页眉页脚 | 基于位置分析 |
| `post_process_md()` 后处理 | 集中清洗逻辑 |
| 目录自动折叠为 `<details>` | 不污染正文检索，保留结构 |
| 动态 `skip_patterns` | 自动匹配 `"文件名·副标题"` 这类品牌信息 |
| 连续相同标题去重 | 解决重复标题问题 |
| 空表格行过滤 | 清理 `\| \| \| \|` 垃圾行 |

### Step 3：向量库从 Chroma 迁移至 FAISS

```python
# 建索引
from langchain_community.vectorstores import FAISS
db = FAISS.from_documents(chunks, embedding)
db.save_local("./faiss_index")

# 加载索引
db = FAISS.load_local("./faiss_index", embedding, allow_dangerous_deserialization=True)
```

### Step 4：接入 CrossEncoder 精排

**检索链路**：召回 8 个候选 chunk → CrossEncoder 打分排序 → 取 top-2 喂给 LLM。

```python
from sentence_transformers import CrossEncoder

@st.cache_resource
def get_reranker():
    """轻量级 CrossEncoder，约 20MB"""
    return CrossEncoder('cross-encoder/ms-marco-MiniLM-L-6-v2')

# 检索时
docs = retriever.invoke(query)  # 粗排召回 k=8
reranker = get_reranker()
pairs = [(query, d.page_content) for d in docs]
scores = reranker.predict(pairs)
ranked = [doc for _, doc in sorted(zip(scores, docs), reverse=True)]
docs = ranked[:2]  # 精排后只取 top-2
```

## 三、关键改动总结

| 文件 | 改动 | 原因 |
|------|------|------|
| `app.py` | `@st.cache_resource` 只缓存 embedding | Chroma 客户端不缓存，避免序列化问题 |
| `app.py` | sidebar 不直接调用 `init_vectorstore()` | 避免每次重渲染触发 HuggingFace 连接 |
| `app.py` | 向量库从 Chroma 切至 FAISS | 纯文件无锁，更稳定 |
| `app.py` | 接入 CrossEncoder 精排 | 减少噪声 chunk 进入 LLM，降低幻觉 |
| `ingest.py` | 换 `pymupdf4llm.to_markdown()` | 保留标题层级、表格结构 |
| `ingest.py` | 添加 `post_process_md()` 后处理 | 去重、过滤、清洗目录 |
