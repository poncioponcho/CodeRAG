## 三、Step 1：清理 docs/ 污染

**zsh 的 `nomatch`**：直接执行 `rm -f docs/*.pdf` 在"无匹配文件"时会触发 `nomatch` 报错并中断，导致清理不完整。

### 3.2 修正后的清理方式


### 3.3 按主题预留子目录（空间隔离）

## 七、Step 5：Hybrid 检索接入 app.py

### 7.1 HybridRetriever 实现

```python
class HybridRetriever:
    def __init__(self, vectorstore, chunks, vec_k=8, bm25_k=8):
        self.vectorstore = vectorstore
        self.chunks = chunks
        # jieba 中文分词构建 BM25
        self.tokenized = [list(jieba.cut(d.page_content)) for d in chunks]
        self.bm25 = BM25Okapi(self.tokenized)

    def invoke(self, query: str):
        # 1. 向量召回（语义相似）
        vec_docs = self.vectorstore.as_retriever(
            search_kwargs={"k": self.vec_k}
        ).invoke(query)
        # 2. BM25 召回（关键词匹配，解决中英文术语鸿沟）
        q_tokens = list(jieba.cut(query))
        scores = self.bm25.get_scores(q_tokens)
        top_idx = np.argsort(scores)[-self.bm25_k:][::-1]
        bm25_docs = [self.chunks[i] for i in top_idx if scores[i] > 0]
        # 3. 合并去重
        seen, results = set(), []
        for d in vec_docs + bm25_docs:
            key = d.metadata.get("source", "") + d.page_content[:60]
            if key not in seen:
                seen.add(key)
                results.append(d)
        return results
```

### 7.2 app.py 检索链路更新

```python
hybrid = HybridRetriever(vectorstore, all_chunks, vec_k=8, bm25_k=8)
# ...
docs = hybrid.invoke(query)
# Step 2: CrossEncoder 精排
reranker = get_reranker()
pairs = [(query, d.page_content) for d in docs]
scores = reranker.predict(pairs)
ranked = [doc for _, doc in sorted(zip(scores, docs), reverse=True)]
docs = ranked[:3]  # 取 top-3
```

## 八、Step 6：评估链路对齐

### 8.1 发现的结构性问题

#### 1）`evaluation_report.json` 仍出现 `README.md`

**原因**：`test_set.json` 里还保留着旧的 `README.md` 来源条目；即使清理了 `docs/README*`，评估依旧会按 `test_set.json` 的 `source` 字段展示为 `README.md`。

**修复**：清理语料/重建索引后，**重新生成测试集**，避免"测试集与语料不一致"。

#### 2）`evaluate_batch.py` 与 `app.py` 检索链路不一致

**问题**：`evaluate_batch.py` 使用 **纯向量** `vectorstore.as_retriever(k=3)`；而前端已经是 **Hybrid（向量+BM25）+ CrossEncoder rerank**。

**修复**：将 `evaluate_batch.py` 的粗召回切换为 Hybrid（向量+BM25），并把 `context` 写成带 `source` 的格式，保证评估更接近线上链路。

#### 3）`generate_test_set_batch.py` 只扫 `docs/*.md` 顶层

**风险**：部分笔记已放进 `docs/dl_nlp/` 等子目录；若测试集生成只看顶层，会导致**生成覆盖不足**、成功率/代表性下降。

**修复**：改为递归扫描 `docs/**/*.md`，并用相对路径作为 `source`（例如 `dl_nlp/transformer-课堂笔记.md`）。

### 8.2 评估结果（Hybrid 对齐后）

```
total_queries: 20
avg_points_coverage: 0.592    (59.2%)
avg_relevance: 3.75
avg_retrieve_ms: 84.5ms
```

**解读**：
- Coverage 大幅提升，说明 Hybrid（尤其 BM25 分支）能把候选拉回"可命中答案要点"的语料区域
- 仍有不少条目的 `accuracy/completeness` 偏低，更多像是**生成/评估口径**问题（问题过细、文档未明确写到、或 LLM 没按"要点"作答），而不是召回完全失效

### 8.3 仍然偏差较大的样本类型

| 问题类型 | 示例 | 根因 |
|----------|------|------|
| 文档未直接包含细节 | "Engram 的 O(1) 哈希检索"、"记忆模块分类" | 应在测试集生成阶段被过滤或降难度 |
| 问题太细/术语需精确定位 | "DuoT5 评估相关性"、"LLMLingua 演示级压缩" | 可能需要更大的候选池，或 rerank 后喂给 LLM 更多 top-n |

## 九、Step 7：测试集过滤与参数调优

### 9.1 测试集生成阶段过滤"不可回答/过细节"

**改动文件**：`generate_test_set_batch.py`

**策略**：弱监督"可回答性"过滤（evidence filter）

- `question` 必须非空
- `answer_points` 必须为非空 list，且长度不超过 6（防止过碎/过细）
- 每个要点截取前 12 个字符作为"短证据片段"，要求在对应文档片段中命中
- 通过阈值：至少命中一半要点（向上取整）

**目的**：减少"文档未直接提及/没有实现细节"造成的评估噪声，让指标更可信。

### 9.2 Hybrid 召回候选池增大 + rerank 后 top_n 提升

| 参数 | 旧值 | 新值 |
|------|------|------|
| `vec_k` | 15 | **25** |
| `bm25_k` | 15 | **25** |
| `rerank top_n` | 3 | **5** |

**目的**：提高召回覆盖与回答完整性，同时依赖 rerank 控制噪声。

### 9.3 互斥运行：batch 脚本运行时禁用前端，前端运行时禁止 batch

**新增**：`core/run_lock.py`

**机制**：本地锁目录 `.coderag_locks/`

| 锁文件 | 由谁维护 | 行为 |
|--------|---------|------|
| `frontend.json` | 前端定期刷新（Streamlit rerun 触发 touch） | — |
| `batch.json` | batch 脚本 acquire + 周期 refresh + finally release | 检测到活跃前端锁 → 直接退出并提示 |

**行为**：
- batch 检测到活跃前端锁会直接退出并提示（避免并行抢 Ollama）
- 前端检测到活跃 batch 锁会进入只读模式（禁用导入/对话入口）

## 十、并发问题诊断

### 10.1 "并行问答 vs batch 生成"导致 epoch 失败

**现象**：前端提问与 batch 生成同时打到同一个 `Ollama /api/generate`，会产生资源竞争（同模型单机并行时更易出现超时、截断、输出非 JSON），从而触发"JSON 解析失败"。

**结论**：**不建议并行**运行网页前端与测试集生成/评估脚本；或至少分离模型/端口/进程。

**缓解**：
- batch 脚本中增加重试与超时（已加入 3 次重试 + timeout）
- 温控/休息保留，提升成功率
- `run_lock.py` 互斥机制强制避免并发
