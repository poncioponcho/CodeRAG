# 2026-04-21｜RAG二次升级：测评正确率优化与检索链路改造

> 目标：在已有 CrossEncoder 精排基础上，解决评估指标全面崩盘问题（覆盖率 6.8%），通过 chunk 策略调优、docs/污染清理、Hybrid 检索（向量+BM25）接入，将要点覆盖率提升至可用水平。
>
> Baseline：覆盖率 6.8% / 准确性 2.1 / 完整性 2.5 / 相关性 2.4

---

## 一、评估指标崩盘诊断

### 1.1 现象

运行 `generate_test_set_batch.py` + `evaluate_batch.py` 后：

| 指标 | 数值 | 判断 |
|------|------|------|
| 要点覆盖率 | **6.8%** | ❌ 严重偏低，检索几乎未召回相关内容 |
| 平均准确性 | 2.1/5 | ❌ LLM 基本在瞎编 |
| 平均完整性 | 2.5/5 | ❌ 答案碎片化 |
| 平均相关性 | 2.4/5 | ❌ 检索片段与问题不匹配 |
| 平均检索延迟 | 80.9ms | ✅ 速度正常 |
| 平均评分延迟 | 5360ms | ⚠️ LLM 评分较慢 |

### 1.2 诊断结论

**不是 LLM "拉了"，是 RAG 管道中间环节断了。**

6.8% 的要点覆盖率说明检索几乎没召回相关内容，LLM 再强也只能瞎编。根因分层：

| 层级 | 问题 | 证据 |
|------|------|------|
| **文档质量层** | docs/ 存在 README.md 广告、空文档、跨主题笔记 | 查询"残差网络"召回 README.md 广告和 XGBoost.md |
| **索引层** | 无关文件污染向量库，稀释召回 | 清理后同一查询仍跑偏 |
| **切分层** | chunk_size=500 把知识点拦腰截断 | 召回的片段不含答案原文 |
| **召回层** | 纯向量对中英文术语鸿沟极弱 | "残差网络"无法命中"ResNet" |
| **精排层** | k=3 召回太少，CrossEncoder 无物可排 | 候选池过小 |

## 二、修复过程

### Step 1：Chunk 策略调优（实验性）

#### 尝试增大 chunk_size（500 → 1200）

**假设**：chunk_size=500 切太碎，一个知识点被拦腰截断，检索回来的是残片。

```python
# 新参数（实验）
splitter = RecursiveCharacterTextSplitter(
    chunk_size=1200,      # 从 500 增大
    chunk_overlap=200,    # 从 50 增大
    separators=["\n\n", "\n", "。", "；", " ", ""]
)
```

**实验结果**：

| 配置 | 覆盖率 | 准确性 | 完整性 | 相关性 | 副作用 |
|------|--------|--------|--------|--------|--------|
| chunk=500（原） | 6.8% | 2.1 | 2.5 | 2.4 | 速度快 |
| chunk=1200（实验） | 9.1% | 2.5 | 2.8 | 3.5 | 速度骤降，生成测试集变慢 |

**问题未解决，且暴露更严重的问题**：chunk=1200 时，查询"残差网络"召回的仍是 README.md 广告和 XGBoost.md，说明**索引污染是首要问题**，单纯改 chunk 救不回来。

**决策：回退 chunk_size 至 500**，优先解决索引污染和召回策略问题。

| 环节 | 实验值 | 回退值 | 原因 |
|------|--------|--------|------|
| chunk_size | 1200 | **500** | 速度恢复，适合技术笔记 |
| chunk_overlap | 200 | **50** | 减少冗余 |
| 粗排召回 | 8 | **15** | 给 Rerank 更多选择 |
| Rerank 输出 | top-2 | **top-3** | 候选更多，答案更完整 |

### Step 3：接入 Hybrid 检索（向量 + BM25）

**根因**：纯向量对"残差网络↔ResNet"这类中英文术语映射极弱，需要 BM25 关键词兜底。

**实现**：

```bash
pip install rank-bm25 jieba
```

```python
from rank_bm25 import BM25Okapi
import jieba
import numpy as np

class HybridRetriever:
    def __init__(self, vectorstore, chunks, vec_k=8, bm25_k=8):
        self.vectorstore = vectorstore
        self.vec_k = vec_k
        self.bm25_k = bm25_k
        self.chunks = chunks
        # jieba 中文分词构建 BM25
        self.tokenized = [list(jieba.cut(d.page_content)) for d in chunks]
        self.bm25 = BM25Okapi(self.tokenized)

    def invoke(self, query: str):
        # 1. 向量召回（语义相似）
        vec_docs = self.vectorstore.as_retriever(
            search_kwargs={"k": self.vec_k}
        ).invoke(query)
        # 2. BM25 召回（关键词匹配）
        q_tokens = list(jieba.cut(query))
        scores = self.bm25.get_scores(q_tokens)
        top_idx = np.argsort(scores)[-self.bm25_k:][::-1]
        bm25_docs = [self.chunks[i] for i in top_idx if scores[i] > 0]
        # 3. 合并去重（向量优先，BM25 补充）
        seen, results = set(), []
        for d in vec_docs + bm25_docs:
            key = d.metadata.get("source", "") + d.page_content[:60]
            if key not in seen:
                seen.add(key)
                results.append(d)
        return results
```

**使用位置**：替换 app.py 中的纯向量 `retriever = vectorstore.as_retriever(...)`。

### 3.1 已实现并验证的（保留）

| 模块 | 状态 | 说明 |
|------|------|------|
| 文档解析 | ✅ 可用 | `ingest.py` 支持多格式转标准 Markdown，PDF 解析质量达标 |
| 向量库 | ✅ 稳定 | 已放弃 Chroma（SQLite 只读锁定），改用 **FAISS**（纯文件，无锁） |
| LLM 调用 | ✅ 稳定 | 原生 HTTP POST `/api/generate`，彻底告别 503 |
| 前端 | ✅ 可用 | Streamlit `app.py` 支持文件上传、对话历史、引用来源展开 |
| 评估框架 | ✅ 可用 | batch 版本能自动跑通，但指标难看 |
| CrossEncoder 精排 | ⚠️ 已加但待验证 | 召回阶段太弱导致无物可排 |

### 3.2 已放弃的路径（不再碰）

| 路径 | 原因 |
|------|------|
| ❌ Chroma | 并发重建时 readonly database |
| ❌ OpenRouter 免费层 | 模型随时下架 / 429 额度耗尽 |
| ❌ `langchain-ollama` 的 ChatOllama | 503 频发 |
| ❌ chunk_size=1200 | 速度骤降，且对短笔记过度切分 |

### 3.3 核心问题定位

> **检索链路断了（不是 LLM 问题）**

1. **索引污染**：docs/ 里有大量无关文件混在向量库里
2. **语义鸿沟**：纯向量对"残差网络↔ResNet"等中英文术语映射极弱
3. **Chunk 策略**：500 字切分导致知识点断裂，需按标题/章节边界切
